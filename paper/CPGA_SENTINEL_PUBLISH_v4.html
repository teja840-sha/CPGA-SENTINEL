Context-Partitioned Generic Assembly with Swarm-External Enforcement for Overcoming LLM Context Degradation

Siva Teja Narayana

Independent Researcher

April 2026

Abstract

LLM compliance with in-context instructions degrades as context grows — a well-documented consequence of finite transformer attention. We present CPGA+SENTINEL, an architecture inspired by genomic assembly pipelines that addresses this through a single principle: classify every context item by whether it genuinely needs the LLM's attention, and externalize everything that does not. CPGA (Context-Partitioned Generic Assembly) partitions context items across four enforcement layers — just as genome assemblers partition sequencing reads into buckets to avoid repeat-induced misassembly. FORGE (Filtered Orchestration for Rule-Governed Execution) is the classification protocol that assigns each item to the cheapest reliable layer: token masks, compiled checkers, the residual prompt, or post-generation judges. SENTINEL (Swarm-Enforced Natural Text Integrity via Networked Evaluation Loops) externalizes enforcement into independent per-item agents organized in four tiers, serving a dual role as both production enforcement mechanism and reusable evaluation suite — the same swarm architecture that enforces compliance during generation constitutes a complete deterministic evaluation harness during benchmarking.

We validate through two complementary evaluations. First, controlled component experiments (7 experiments, 24 custom constraints) isolate each primitive's contribution: CADG constraint-order permutation yields +8pp at 10 constraints; FORGE filtering improves compliance on residual (Layer 3) rules by +11–18pp by concentrating attention on them; SENTINEL achieves constant prompt cost (57 tokens regardless of rule count). Second, public benchmark validation on three public benchmarks (MOSAIC, IFBench, IFEval) and one custom benchmark (SysPrompt, designed for this work) totaling 1,081 evaluation instances across 104 constraint types, all scored by deterministic Python verifiers with bootstrap 95% confidence intervals: the best-performing composition achieves +6.4pp on MOSAIC (Full Stack, 0.758→0.822), +9.2pp on IFBench (CADG+SENTINEL, 0.397→0.489), +7.6pp on SysPrompt (CADG+SENTINEL, 0.862→0.938), and +5.8pp on IFEval (Full Stack, 0.799→0.857), all substantially exceeding a strong retry-with-feedback baseline. Component composition varies by benchmark characteristics — FORGE's value scales with rule taxonomy complexity (§6.9.4). In production deployment on structured content generation (577 rules, 248K chars reference context), the architecture achieves a 16× context reduction, 100% structural compliance, and +14.3pp quality improvement over the prior system.

Keywords: context degradation, LLM compliance, instruction following, prompt engineering, agent orchestration, swarm enforcement

1. Introduction

1.1 The Context Degradation Problem

Every production LLM application loads context into the prompt: instructions, constraints, reference documents, tool definitions, conversation history, examples, schemas, and behavioral guidelines. As applications mature, this context grows. The problem is universal:

  • Agent instruction files: A 2,000-line agent instruction file guides a complex workflow. By line 800, the agent skips steps and hallucinates task completion. Instructions at positions 40–60% fall in the attention dead zone (F2).

  • System prompts with behavioral guidelines: A bot with 50 behavioral rules systematically under-follows rules #20–#35 (the middle of the list). Adding rule #51 degrades compliance on all existing 50 (F4).

  • RAG retrieval context: Of 20 retrieved documents, positions 1–3 and 18–20 influence the answer strongly; positions 8–12 are functionally ignored (F2). Adding more documents dilutes attention across all of them (F1).

  • Code generation with specifications: Architecture docs (8K tokens, early position) and the task (500 tokens, end position) are followed; a style guide (3K tokens, middle position) is silently ignored. Those tokens consumed attention budget without producing compliance.

In production systems we have observed, instruction contexts routinely grow to 50,000–300,000+ characters of combined skill instructions, reference documentation, behavioral guidelines, and schema definitions — all competing for the model's attention. This is not a contrived scenario; it is the natural end-state of any LLM application that accumulates requirements, tools, and reference material over its operational lifetime.

The core insight: every token of context you add to an LLM prompt makes every other token of context slightly less effective. This is not a prompting failure, a model limitation to be fine-tuned away, or a training data gap. It is a mathematical property of softmax attention: the attention budget is finite, and adding context dilutes it.

1.2 Why Current Approaches Fail

The transformer's self-attention mechanism (Vaswani et al., 2017) distributes attention across all tokens via softmax normalization. Adding context tokens dilutes the attention available to existing tokens — not by a fixed ratio (the distribution depends on content), but monotonically. The empirical consequence is well-documented: more context items in the prompt means less reliable adherence to any individual item.

Six independent studies published between 2024–2026 quantify the resulting failures. Context Rot [2] shows 30% accuracy drops at ~113K tokens across 18 frontier models. Lost in the Middle [3] measures 30%+ drops as relevant information moves to position 10 of 20. MOSAIC [4] — the most directly relevant — shows compliance dropping from 77.67% (few constraints) to 32.96% (many), with GPT-4.1 scoring 12.6% at 800 rules. Effective Window Collapse [9] finds 128K-token windows yield only 20–40K effective tokens for instruction following. Position bias studies [15] document systematic U-shaped attention (strong at edges, weak in the middle 40%). Reasoning Diversion [6] shows that chain-of-thought actually degrades instruction compliance across 15 models by diverting attention from constraint tokens to reasoning tokens. Section §2 formalizes these findings as failure modes F1–F12.

The transformer architecture has a fundamental context capacity ceiling. No prompting technique or chain-of-thought strategy has overcome this ceiling (F7 shows CoT can worsen it). Fine-tuning bypasses the context window but does not scale to hundreds of frequently-changing rules. The alternative is architectural: reduce what goes into the context, and enforce what can be enforced outside it.

1.3 The Genomic Assembly Analogy

The initial motivation for this work came from an analogy with computational genomics. The Bucket-Order-Assemble (BOA) framework used in de novo genome assembly faces a structurally similar problem: when assembling a genome from millions of short sequencing reads, repetitive DNA structures cause misassemblies. The reads from different copies of a repeat are indistinguishable, and the assembler incorrectly joins sequences that should be separate.

The BOA solution: partition sequencing reads into smaller, manageable buckets based on k-mer similarity, assemble each bucket independently in parallel, and combine results deterministically using overlap graphs. This divide-and-conquer approach reduces misassemblies because each bucket contains reads from only one copy of the repeat.

The parallel to LLM context management is instructive, though the underlying mechanisms differ (k-mer overlap graphs vs. softmax attention budgets):

Genomic Assembly LLM Context Management
Repetitive DNA structures cause misassembly Competing context items cause attention failures
Reads from different repeats are indistinguishable Instructions from different sources compete for attention
Partition by k-mer similarity Partition by context type and enforcement mechanism
Assemble each bucket independently Process each context partition independently
Combine via deterministic overlap graph Combine via deterministic orchestration
Misassembly rate drops with bucket isolation Performance rises with context isolation

This led to the central question: what if we stopped treating the LLM's context window as a reliable container for everything the system needs to know, and instead engineered the pipeline so that each piece of context is enforced through the cheapest mechanism that can handle it — with only the genuinely hard parts remaining in the LLM's attention?

2. Failure Mode Taxonomy

We catalog twelve failure modes across three categories, synthesized from six independent studies published 2024–2026. Each failure mode is assigned a code (F1–F12) and mapped to the architectural primitive(s) that neutralize it. This taxonomy provides a structured vocabulary for discussing why LLMs degrade as context grows, which informs where in the architecture each failure should be addressed. While the cited studies focus on instruction-following and rule compliance, the underlying mechanisms (attention dilution, positional bias, effective window collapse) affect all context in the prompt — not just rules.

2.1 Context-Length Failures (F1–F3)

These failures arise from the finite capacity of transformer attention. They affect all context — instructions, documents, tool definitions — not just rules.

Code Name Source Quantified Impact Mechanism
F1 Context Rot Chroma (2025) 30% drop at ~113K tokens; 18/18 models affected Self-attention dilution as context grows; no model exempt
F2 Lost in the Middle Liu et al. (TACL 2024) 30%+ drop at position 10/20 U-shaped attention: strong at edges, 40–60% is a dead zone
F3 Effective Window Collapse Zylos (2026) 128K advertised → 20–40K effective Models break at 30–40% of advertised limit for IF tasks

Architectural implication: Any context item in the middle of a large prompt is systematically under-attended. An architecture that loads everything into one prompt inherits this failure regardless of model capability.

2.2 Instruction-Following Failures (F4–F7)

These failures are specific to instruction-following contexts, where the number and arrangement of context items directly affects how well any individual item is attended to.

Code Name Source Quantified Impact Mechanism
F4 Rule Count Degradation MOSAIC (EACL 2026) 77.67% → 32.96% compliance; 12.6% at 800 rules Each additional rule dilutes attention to all existing rules
F5 Position Bias ACL 2025 Middle 40% of rules systematically under-followed Primacy/recency effects in attention allocation
F6 Constraint Interaction MOSAIC (EACL 2026) Synergistic pairs help (+); antagonistic pairs hurt (−) Rules can reinforce or undermine each other's compliance
F7 Reasoning Diversion arXiv 2505.11423 CoT degrades IF across 15 models Chain-of-thought tokens compete with constraint tokens

Architectural implication: Adherence to any single item is a function of all items in the prompt. Reducing the count improves adherence to the remainder even with no other change.

2.3 Agent Workflow Failures (F8–F12)

These failures occur when LLMs are used as agents in multi-step workflows.

Code Name Source Mechanism
F8 False Termination OpenReview 2025 Agent completes 3/12 steps, declares "done" with confidence
F9 Step Skipping AgentHallu (2025) Out-of-order execution, omitted intermediate steps
F10 Hallucinated Delegation AgentHallu (2025) Orchestrator claims delegation but generates output itself
F11 Cascade Propagation AgentHallu (2025) Error in step N propagates through all subsequent steps

Architectural implication: Workflow orchestration must be deterministic (code, not LLM); state transitions must be verified programmatically.

3. CPGA Architecture: Four-Layer Enforcement

3.1 Design Principle

Existing systems typically apply one or two enforcement mechanisms: everything in the prompt (monolithic), or prompt plus a single external validator (e.g., Guardrails AI's retry loops, structured output schemas). Some constrain decoding (Outlines, llguidance). Each approach treats context items as belonging to a single enforcement category.

We observe that context items are not homogeneous. "Do not use the × symbol" and "ensure the argument is logically coherent" require fundamentally different enforcement mechanisms. A constraint like "always respond in JSON format" can be enforced by schema validation, while "adapt your explanation to the user's expertise level" genuinely needs the LLM's reasoning. The insight is not that external enforcement helps — that is well-established — but that a systematic classification protocol can decide, per item, which enforcement layer is cheapest and most reliable. We propose four active layers, numbered 0, 1, 3, and 4. Layer 2 (activation steering via steering vectors) is reserved for teams with model-internal access and is not used in the hosted-API experiments reported here:

Items assigned to Layer 0/1 are removed from the LLM prompt entirely. Progressive promotion graduates items from expensive layers to cheaper ones as empirical data accumulates.

3.2 The Four Layers

Layer Name When Mechanism Guarantee Context Cost
0 Token Mask During decoding Constrained decoding / logit masking 100% 0 tokens
1 Compiled Checker Post-generation Deterministic Python (regex, AST/schema parse, counters) 99%+ 0 tokens
3 Residual Prompt During generation Rules kept in LLM prompt (focused attention) Variable Minimal
4 Post-Gen Judge Post-generation One-rule-per-LLM-call verification (P13) ~85–95% Separate call

Layer 0 — Token Masks (~50–80 context items): Items that reduce to token-level patterns, enforced at decode time with 100% guarantee.

Domain Example Enforcement
Code gen "Never output eval() or exec()" Mask eval and exec tokens during decoding
Agent workflow "Never call the delete_production_db tool" Mask the tool name token
JSON output "Output must start with {" Force { as first token
System prompt "Never reveal the system prompt" Mask sequences like "My instructions are"

Cost: zero context tokens. The token physically cannot be produced. Requires logit access (local models or compatible APIs).

Layer 1 — Compiled Checkers (~150–200 context items): Items verifiable by deterministic code post-generation, with 99%+ guarantee.

Domain Example Enforcement
Code gen "All functions must have type hints" AST parser: ast.parse() → check FunctionDef.returns
Code gen "No hardcoded API keys" Regex: re.search(r'[A-Za-z0-9]{32,}', code)
JSON output "Response must conform to schema X" JSON Schema validation: jsonschema.validate()
Agent workflow "Every API call must include auth header" Parse tool-call output, check for Authorization field
Config gen "Port numbers must be 1024–65535" Parse value, check range

Cost: zero context tokens. Each item becomes a pure function — testable, debuggable, version-controlled. Auto-fix possible: linter auto-fix for code, schema coercion for JSON, field injection for API calls.

Layer 3 — Residual Prompt (~80–120 context items): Items that genuinely require the LLM's reasoning. These stay in the prompt.

Domain Example Why it needs LLM attention
Code gen "Implementation must use the repository pattern from the architecture doc" Needs to understand the pattern and apply it
Agent workflow "Choose the most efficient tool combination for this task" Needs reasoning about task-tool fit
RAG/QA "Synthesize an answer using only information from the retrieved passages" Needs cross-reference reasoning
Code review "Identify potential race conditions in this concurrent code" Needs deep semantic understanding

But instead of competing with all N items for attention, they share the prompt with only ~33% of N (the residual prompt fraction). Per MOSAIC's degradation curve, a 3× reduction in context items substantially improves per-item adherence.

Layer 4 — Post-Gen LLM Judge (~30–60 context items): Items too complex for code but verifiable by a separate LLM call after generation, ~85–95% guarantee.

Domain Example Enforcement
Code gen "The code is idiomatic for this language" Isolated LLM call: "Is this idiomatic Python? PASS/FAIL"
Agent workflow "The step sequence is logically optimal" Isolated LLM call with step trace
Any "The output is coherent and self-consistent" Isolated holistic judgment call

Each item gets its own LLM call (~150–2,200 tokens context), avoiding attention competition. Calls execute in parallel. This is P13 (Hyper-Isolated Verification).

3.3 Automated Classification Protocol

A critical question: how does the system decide which layer each context item belongs to? This cannot be manual — production systems have hundreds of items. The classification protocol has three phases: probe, validate, and promote.

Phase 1: Probe (LLM-assisted, runs once per item at setup)

For each context item, an LLM classifier answers a sequence of questions. The classifier runs once at system setup — not per generation.

INPUT: A context item in natural language.

PROBE 1 — TOKEN BAN TEST:
  Ask: "Does this item ban or require a specific word, symbol, 
        or character that can be identified as a single token?"
  If YES: Extract the token(s). 
  → Candidate: LAYER 0

PROBE 2 — CODE CHECKER TEST:
  Ask: "Can you write a Python function that takes the LLM's output 
        as a string and returns True if this item is satisfied, 
        False otherwise? If yes, write the function."
  If the LLM produces a function:
  → Candidate: LAYER 1

PROBE 3 — REASONING REQUIREMENT TEST:
  Ask: "Does satisfying this item require the model to reason about 
        the content during generation — understanding context, making 
        decisions, applying knowledge? Or can it be verified after?"
  If needs reasoning during generation:
  → LAYER 3 (residual prompt)
  If can be verified after:
  → LAYER 4 (post-gen judge)

Key property: The probes are hierarchical. An item assigned to Layer 0 never reaches Probe 2. This prevents misclassification — a token ban is also code-checkable, but Layer 0 is cheaper, so it wins.

Phase 2: Validate (deterministic, runs on test corpus)

Classification is not trusted on the LLM's word alone. Each classification is validated:

LAYER 0 VALIDATION:
  - Tokenize the banned token(s) with the model's tokenizer
  - Confirm each is a single token or short sequence
  - Test: generate 10 outputs WITH the mask active
  - Verify: the banned token never appears
  → If any test fails: demote to Layer 1

LAYER 1 VALIDATION:
  - Run the generated checker function on 10+ known-good outputs
    (should return True for all)
  - Run on 10+ known-bad outputs (should return False for all)
  - Measure precision and recall
  → If precision < 95% or recall < 90%: demote to Layer 3
  → If function throws exceptions: demote to Layer 3

Items that fail validation at every layer default to Layer 3 (residual prompt). This is safe — the worst case is that the item stays in the prompt where it would have been anyway in a traditional system.

Phase 3: Promote (continuous, runs after every production batch)

After each production run, the system checks for promotion opportunities:

FOR EACH LAYER 3 ITEM (in prompt):
  - Did the post-gen checker (Layer 1 candidate) agree with 
    the item's pass/fail status in 95%+ of cases?
  → YES: Promote to Layer 1. Remove from prompt.

FOR EACH LAYER 4 ITEM (LLM judge):
  - Does a simpler heuristic agree with the judge 95%+ of cases?
  → YES: Replace judge with heuristic. Promote to Layer 1.

FOR EACH LAYER 1 ITEM (compiled checker):
  - Can the checker be reduced to a single-token ban?
  → YES: Promote to Layer 0.

The system self-improves: Layer 3 shrinks with every batch. More items get enforced by code, fewer by the prompt. The LLM's attention concentrates further on the genuinely hard items.

Human-in-the-loop gate: In our implementation, promotions are not automatic. When a rule reaches the promotion threshold, the system creates a pending notification. A human reviews the recommendation and explicitly approves or skips. This prevents premature promotion of rules where the compiled checker is subtly incorrect or where the high pass rate reflects a lenient checker rather than genuine model compliance.

Classification Example

To illustrate the hierarchy: "Never generate eval() or exec()" is caught at Probe 1 (single-token ban → Layer 0); "All functions must have type hints" passes Probe 2 (AST checker → Layer 1); "Use the repository pattern from the architecture doc" requires in-generation reasoning (Layer 3); "Output is idiomatic" can only be judged post-hoc (Layer 4). Each item is classified to the cheapest reliable tier.

Cost and Distribution

Classification runs N LLM calls at setup (one per item, ~200 tokens; total ~$0.001 × N). Validation runs generated checkers on a test corpus — pure Python, near-instant. Observed distribution across production systems: ~12% Layer 0, ~45% Layer 1, ~33% Layer 3, ~10% Layer 4 — a 3–4× prompt reduction with higher enforcement reliability than the prompt provided.

3.4 Progressive Promotion Protocol

The layer assignments are not static. The system improves with every production run:

  • A Layer 3 rule (in prompt) that a compiled checker could have caught → generate the checker, validate on 10+ examples, promote to Layer 1. The prompt loses one rule; attention concentrates further.
  • A Layer 4 rule (LLM judge) where the judge's verdict agrees with a regex heuristic 95%+ of the time → the heuristic is the rule. Promote to Layer 1.
  • A Layer 1 rule (checker) that can be reduced to a single-token pattern → promote to Layer 0.
  • Result: Layer 3 (the prompt) shrinks with every batch. The LLM's attention gets more concentrated. Compliance on residual rules increases. Cost decreases. The system gets better without model changes.

4.2 Detailed Primitive Descriptions

P10: Intermediate Representation (Layers 0+1). The LLM outputs typed JSON conforming to a strict schema. A deterministic compiler produces the final artifact using production formatters. Since structural invariants are enforced in Python code — literal heading text (hardcoded), required elements (always emitted), section ordering (assembler rejects violations) — they cannot be violated regardless of LLM output. The set of valid artifacts is a strict subset of valid IRs composed with the compiler. This is the genomic assembly insight made concrete: the LLM is the sequencer (creative, noisy), the compiler is the assembler (deterministic, correct).

P1: Scope Partitioning (Layer 3). Items within the residual prompt are decomposed by scope into logically grouped subsets. For N total items, M scopes, G global items: per-agent load = N/M + G. For example, with N=300 residual items, M=6 scopes, G=50 global: per-agent = 100 items (vs 300), a 3× reduction. Our experiments (§6.2) show that the method of partitioning matters critically — generating per-scope and merging hurts, but grouping constraints by category within one prompt helps.

P3 + P4: Competitive Generation with Deterministic Scoring (Layer 3). Generate N independent candidates and select the best using deterministic checkers. For single-agent compliance probability p, N independent candidates: P(rule followed) = 1 − (1−p)^N.

p (single agent) N=1 N=2 N=3 N=5
0.70 0.700 0.910 0.973 0.998
0.80 0.800 0.960 0.992 0.99968
0.90 0.900 0.990 0.999 0.99999

Critical caveat: This formula assumes failure independence across candidates. Our experiments (§6.3) confirm failures are often correlated — the same constraints fail in all candidates when the model systematically misinterprets them. CADG (§4.3) addresses this.

P13: Hyper-Isolated Verification (Layer 4). Instead of evaluating all rules in one LLM call (where rules compete for attention), evaluate one rule per call. Each call receives ~150–2,200 tokens of context: the generated text + a single rule. Attention decay is negligible at this context size. Calls execute in parallel. Our experiments (§6.8) show this requires a frontier-class model (Claude Opus 4.6 or GPT-5.4) to achieve reliable results.

P6: Targeted Mutation (Layer 3). When competitive generation does not fully fix a failure, a repair agent receives ONLY: the failing section (~1–3K tokens) + the failed rules with evidence (~0.5–2K tokens) + adjacent context (~1–2K tokens). Total context: ~3–7K tokens. After K rounds with fix rate f per round: remaining failures = initial × (1−f)^K. For f=0.6, K=3: 6.4% remain. For f=0.7, K=3: 2.7% remain.

4.3 CADG: Constraint-Aware Diverse Generation

Standard competitive generation (P3) generates N candidates from the same prompt. Our experiments revealed that failures are often correlated: if the model misunderstands a constraint, all N candidates fail it identically. The effective N becomes 1.

CADG engineers diversity to de-correlate failures through four mechanisms:

Mechanism 1 — Constraint Permutation. Reorder constraints differently for each candidate. MOSAIC proved position bias (F5) exists — early rules get primacy attention. By rotating which rules get the primacy slot:

Candidate 1: [C1, C2, C3, ..., C20]     → C1 gets primacy
Candidate 2: [C11, C12, ..., C20, C1...] → C11 gets primacy  
Candidate 3: [C7, C8, ..., C20, C1...]   → C7 gets primacy

Cost: zero (just reorder the prompt). Each candidate sees the same rules but in a different order, ensuring different rules benefit from the primacy attention slot.

Mechanism 2 — Constraint Emphasis Rotation. For each candidate, emphasize a different subset of constraints (bold, uppercase, repeated). Each candidate "specializes" in different rule groups. Cost: zero (prompt modification only).

Mechanism 3 — Model Rotation. Use different frontier models per candidate (Claude Opus 4.6, GPT-5.4, Gemini 2.5 Ultra). Model-specific blind spots are uncorrelated across model families. Cost: same token budget, requires multiple API keys.

Mechanism 4 — Reformulation Diversity. Rephrase context items differently per candidate while preserving meaning. "All functions must have type hints" → "Every function definition requires return type and parameter type annotations" → "Type annotations are mandatory on all function signatures." The model's interpretation depends on exact wording; varying it de-correlates interpretation-based failures.

5. SENTINEL: Swarm-External Enforcement

5.1 From In-Prompt to External Enforcement

The principle of separating generation from enforcement is not new — DCCD (2026) drafts unconstrained then applies constrained decoding, and Guardrails AI (2025) validates then retries. SENTINEL extends this to a multi-tier swarm where each context item gets its own independent enforcement agent, operating at the tier (token, chunk, document, or semantic) appropriate to that item's complexity:

BASELINE:   [N rules + task] → LLM → [output following ~33% of rules]

SENTINEL:   [task only]      → LLM → [clean creative output]
                                         ↑ ↑ ↑ ↑ ↑
                                         S₁ S₂ S₃ S₄ S₅ ... Sₙ sentinels

The LLM sees zero enforcement context — its entire attention budget goes to the creative/reasoning task. Enforcement-related context degradation (F1–F7, §2) is structurally mitigated: no enforcement items in the prompt means no context rot (F1), no lost-in-the-middle (F2), no count degradation (F4), no position bias (F5), no cross-item interference (F6), and no reasoning diversion (F7). Each sentinel operates independently outside the model's context window. (Note: the task prompt itself still consumes attention; SENTINEL addresses enforcement context, not task complexity.)

5.2 Four-Tier Sentinel Architecture

Sentinels are classified by when and how they act:

Tier Scope Typical Rules Mechanism Guarantee Latency
1 — Token Per token ~50–80 Character/symbol masks before sampling 100% Decode speed
2 — Chunk Per sentence/paragraph ~150–200 Regex, word count, pattern detection 99%+ Microseconds
3 — Document Complete output ~100–150 AST/schema parse, cross-reference, structural validation 95%+ Milliseconds
4 — Semantic Complete output ~50–100 Micro-LLM call (~150 tokens context, parallel) ~85% ~1–2 seconds

Each sentinel is an independent process with a minimal interface:

class Sentinel:
    item_id: str           # e.g., "no_eval_exec"
    item_text: str         # e.g., "Never generate eval() or exec()"
    check: Callable        # e.g., lambda code: "eval(" not in code and "exec(" not in code
    fix: Callable | None   # e.g., lambda code: code.replace("eval(", "safe_eval(")
    tier: str              # "token" | "chunk" | "document" | "semantic"

    def watch(self, output: str) -> Verdict:
        if self.check(output):
            return Verdict.PASS
        if self.fix:
            return Verdict.FIX(self.fix(output))
        return Verdict.FLAG(self.item_text)

5.3 The Arbiter

Collects verdicts from all sentinels and determines action:

  1. All PASS → ship the output
  2. Some FIX → apply fixes sequentially, re-check affected sentinels, ship if clean
  3. Some FLAG (no auto-fix available) → group flagged rules by section. Send only the flagged section and only the flagged items to a repair LLM. The repair LLM sees 3–5 items, not the full N. Repair output goes back through the swarm for re-verification. Max 3 repair cycles.
  4. Persistent failures → escalate to human with full sentinel report

5.4 Progressive Promotion (The Evolver)

The sentinel swarm gets smarter over time:

  • A Tier 4 sentinel (semantic, LLM-based) that consistently agrees with a regex pattern → auto-generate the regex → promote to Tier 2. One fewer LLM call per generation, forever.
  • A new failure pattern found by human QC → generate a new sentinel. One more watcher, permanently.
  • Sentinels that never fire (rule never violated) → deprioritize. Save compute cycles.
  • Sentinels that conflict (fixing one breaks another) → detected by the Arbiter, surfaced to human for rule resolution.

Over time, the swarm evolves toward more Tier 1–2 sentinels (cheap, fast, code) and fewer Tier 4 sentinels (expensive, LLM-based). The system's cost and latency decrease while its reliability increases.

Adaptive rule generation: The swarm also grows. When the same failure pattern occurs repeatedly (≥5 occurrences across ≥3 different inputs), the system generates a candidate sentinel — deterministic (Layer 1) if a code checker can be synthesized, semantic (Layer 4) otherwise. In our implementation, candidate rules require explicit user approval before activation, preventing the system from learning incorrect patterns from noisy failure data.

5.5 Sentinel Factory: Dynamic Sentinel Construction

Sentinels are not hardcoded — they are dynamically constructed from FORGE classification output. The Sentinel Factory consumes two artifacts produced by FORGE:

  • layer1_compiled.json (FORGE Layer 1 output) → Tier 2/3 deterministic checkers (compiled Python functions). Each rule classified as programmatically enforceable is compiled into a standalone checker function.

  • layer4_judges.json (FORGE Layer 4 output) → Tier 4 semantic verification prompts. Each rule requiring LLM judgment is packaged as an isolated P13 verification prompt.

Each compiled checker undergoes a sanity test — it must not crash on empty input — before activation. Deduplication against existing hardcoded sentinels prevents double-checking: if a manually authored sentinel already covers a rule, the factory-generated version is suppressed.

This design means that when source documents change, FORGE reclassifies rules, and the Sentinel Factory automatically builds new sentinels. No manual sentinel authoring is required. This is the CPGA principle applied to its own enforcement layer: the enforcement suite is assembled from the same classification that partitions the generation context.

5.6 Content Sentinel Taxonomy

In production deployment, the sentinel swarm comprises individual checks organized into domain-relevant categories. Each check is executed as a P13 isolated LLM call with approximately 150–300 tokens of context. When a check returns FAIL, it provides: (1) specific failure evidence (quoted text from the output), and (2) a fix instruction for the mutation agent. Some checks require full-output context (e.g., cross-section coherence); most operate on individual sections. Some checks additionally require specification attributes — for example, verifying vocabulary appropriateness requires knowing the target audience profile.

The taxonomy follows a universal seven-category framework applicable to any structured content domain:

Category Typical Checks Example (Code Gen) Example (Content Gen) Scope
A. Factual Accuracy 3–5 "Do all unit tests pass the stated assertions?" "Verify every computation independently" Per section
B. Quality & Best Practices 5–8 "Does each function follow single-responsibility principle?" "Does each concept have a concrete example before generalization?" Per section
C. Language & Audience Fit 3–5 "Are variable names descriptive and convention-compliant?" "Are all terms appropriate for the specified audience level?" Per section + spec
D. Structural Integrity 3–5 "Does the module export only the declared public API?" "Does the parallel structure maintain consistent element count?" Per section
E. Specification Compliance 3–5 "Does the implementation satisfy all interface contracts?" "Does the output follow the required instructional sequence?" Full output
F. Requirements Alignment 2–4 "Are all acceptance criteria from the ticket addressed?" "Are deliverables aligned with the input specification?" Full output + spec
G. Coherence & Flow 2–4 "Do module dependencies form a DAG with no circular imports?" "Do transitions between sections maintain logical progression?" Full output

As a reference implementation, we define 31 sentinel check templates across these 7 categories; a production deployment may instantiate 153+ concrete sentinel instances from these templates depending on the rule taxonomy (§6.10). The framework is domain-agnostic: deployers define their own category-specific checks as P13 prompts. The infrastructure — parallel execution, verdict collection, arbiter aggregation, fix routing — remains identical regardless of domain.

5.7 Multi-Source Sentinel Growth

The sentinel swarm grows its repertoire from three independent, complementary sources — ensuring that the enforcement suite captures rules that no single extraction method could find alone:

  1. FORGE classification of source documents (§5.5) — captures documented requirements.
  2. Adaptive rule generation from recurring failures (§5.4) — captures emergent requirements not in source documents (≥5 occurrences across ≥3 inputs triggers synthesis).
  3. Human expert feedback distillation — captures tacit expert knowledge from quality review patterns, formalized as sentinel rules with explicit approval gates.

The repertoire is self-expanding: every production run may trigger source (2), every review cycle may contribute via source (3). Enforcement capability increases monotonically with use.

5.8 SENTINEL as Generalizable Evaluation Suite

The SENTINEL swarm serves a dual role: enforcement during production AND evaluation during development and benchmarking. The same swarm architecture — independent per-rule agents, deterministic scoring at Tiers 1–3, isolated LLM verification at Tier 4 — constitutes a complete evaluation suite when deployed without the auto-fix and repair loops.

In evaluation mode:

  • Tier 2 checkers (regex, pattern detection) → deterministic pass/fail per constraint, identical to benchmark scoring functions.

  • Tier 3 validators (structural, cross-reference) → document-level quality metrics.

  • Tier 4 judges (P13 isolated calls) → semantic quality assessment, one check per call.

The public benchmark experiments (§6.9) directly instantiate this pattern: each benchmark's native scoring functions are loaded as Tier 2/3 sentinels, and the SENTINEL arbiter aggregates verdicts. The architecture does not distinguish between "running MOSAIC's evaluation" and "running the production sentinel swarm" — they use the same execution engine.

This has a practical consequence: any organization deploying CPGA+SENTINEL for production enforcement automatically has a regression testing suite. Adding a new sentinel for enforcement simultaneously adds a new evaluation check. The eval suite grows with the enforcement suite, and both share the same deterministic scoring infrastructure.

6.10.2 FORGE Classification in Production

FORGE classifies 577 rules across four enforcement layers:

  • ~68 rules → Layer 0 (token masks): Enforced at decode time via token suppression. Includes banned symbols, required/forbidden characters, and format-level token constraints.
  • ~132 rules → Layer 1 (compiled checkers): 0 tokens in prompt, 99%+ enforcement guarantee. Includes formatting rules, structural element presence, attribute validation, and rendering constraints.
  • ~277 rules → Layer 3 (residual prompt): partitioned by section scope, ~15 rules per section agent. Includes content quality rules, domain-specific language requirements, and explanation quality.
  • ~100 rules → Layer 4 (post-gen judges): 1 rule per isolated LLM call. Includes holistic quality assessments, cross-section coherence, and output appropriateness.

FORGE's production value comes from the compound effect: removing 200+ rules from the prompt allows the LLM to concentrate attention on the remaining ~15 rules that genuinely require reasoning. The production pipeline's rich rule taxonomy provides significantly more classification opportunities than the flat constraint lists in synthetic benchmarks — explaining why FORGE's measured contribution is modest on MOSAIC (+0.9pp) but substantial in production (+14.3pp as part of the full stack).

6.10.5 Self-Improvement Mechanisms in Production

The production deployment validates three self-improvement mechanisms that were theoretical in the controlled experiments:

  • Progressive promotion (§3.4): After each production batch, SENTINEL tracks per-rule pass/fail rates. Rules that pass 95%+ over 10+ observations are candidates for promotion from Layer 3 (prompt) to Layer 1 (compiled checker). All promotions require explicit user approval. The prompt shrinks with every batch.
  • Sentinel feedback loop to FORGE (§9.3): Rules that pass unprompted (sentinel passes when the rule was not in the section agent's prompt) provide zero-cost promotion signals. When a rule reaches 95%+ unprompted pass rate over 10+ observations, it is flagged for promotion.
  • Adaptive rule generation: Recurring failures (≥5 occurrences across ≥3 inputs) trigger automatic generation of new sentinel rules. Deterministic (regex) rules are preferred; semantic (P13) rules are generated as fallback. All new rules require user approval.

8. Related Work

The individual techniques in CPGA+SENTINEL draw from established research threads: intermediate representations from compiler design, best-of-N selection from sampling theory, deterministic orchestration from workflow engines, and post-generation validation from multi-agent systems. We do not claim novelty for these individual components. The contributions are: (a) the FORGE classification protocol for systematically deciding which enforcement mechanism handles each context item, (b) CADG's exploitation of position bias as a diversity mechanism, (c) progressive promotion as a self-improving feedback loop, and (d) the integrated composition with experimental evidence. We position each component against its closest prior art below.

8.1 Constrained Generation

Outlines/llguidance (2024–2025): Enforce grammar at the token level via logit masking. Maps to our Layer 0. CPGA's contribution: classifying which rules belong at Layer 0 vs higher layers; most rules cannot be token-masked.

ATLAS-RTC (arXiv 2603.27905, 2026): Graduated token control for structured outputs (JSON, XML). Handles structural rules but not semantic content rules. Maps to our Layers 0–1. CPGA handles the full spectrum through four layers.

DCCD (March 2026): Draft with unconstrained generation, then apply constrained decoding. Closest to SENTINEL's "generate clean, enforce externally" principle. Key differences: DCCD is structural only (token/grammar level); SENTINEL operates at four tiers including semantic. DCCD has no swarm, no auto-fix, no progressive promotion.

8.2 Multi-Agent Compliance

Guardrails AI (2025): Sequential validators with retry loops. SENTINEL's contribution: concurrent per-rule agents (not sequential) organized in four tiers with auto-fix capabilities. Guardrails validators run serially and do not fix — they retry the entire generation.

interwhen (Microsoft 2026): Monitors reasoning traces for logical errors. Single monitor, not a swarm. Targets reasoning correctness, not multi-rule content compliance.

8.3 Competitive Generation

RoBoN (arXiv 2512.05542, 2025): Routes across models based on reward scores. Uses the same prompt for all models — no constraint permutation. CADG adds principled input diversity through four de-correlation mechanisms.

FUSION (ICLR 2026): Synthesizes from multiple candidates via LLM fusion. The fusion LLM has its own attention budget and can drop constraints during synthesis. CADG selects rather than fuses, using deterministic scoring.

ModeX (arXiv 2601.02535, 2025): Evaluator-free selection via semantic consensus — picks the MODE (most common output). For compliance, the mode is the correlated failure; CADG picks the outlier that passes the hard constraints.

Wang et al. (2025): Diverse sampling for best-of-N. Shows temperature and sampling diversity help. CADG goes further: diversity through prompt structure (permutation, emphasis), not just sampling parameters.

8.4 Context Management

CoDA (WSDM 2026): Decouples planner and executor context. Isolates task context across subagents but loads the full rule set into each. CPGA partitions both task and rules.

MetaGlyph (2025): Symbolic prompt compression. Compresses all rules equally. FORGE removes rules entirely — cheaper and more effective than compression.

RLM (MIT 2025): Recursive context exploration for long documents. Handles long inputs, not large rule sets.

8.5 Intermediate Representations and Structured Output

JSON-mode / Structured Outputs (OpenAI 2024, Anthropic 2025): API-level enforcement of output schema conformance. Maps to our Layer 0–1 for structural constraints. CPGA extends this by classifying which constraints CAN be schema-enforced vs. which require semantic enforcement at higher layers.

Compiler IR (standard): The principle of separating a high-level representation from its final rendering is foundational in compiler design (LLVM IR, AST-based code generation). P10 applies this to LLM outputs: the model produces a typed intermediate form, and a deterministic compiler handles formatting invariants. The novelty is not the IR concept but its application to LLM compliance — making structural violations unrepresentable in the output schema.

G-Eval (Liu et al., 2023) and FActScore (Min et al., 2023): Decompose complex evaluation into atomic judgments. P13 applies the same insight to compliance verification — one context item per LLM call. The contribution is integrating this into a tiered enforcement architecture rather than using it standalone for evaluation.

8.6 Rule Extraction and Management

ConInstruct (2025): Detects constraint conflicts at 91% F1. FORGE incorporates conflict detection at CLASSIFICATION time — before generation, not during.

AutoRule (2025): Extracts rules from user preferences. Rules stay as natural language for LLM judge. FORGE compiles rules to deterministic code (Layer 1) wherever possible.

AutoQual (EMNLP 2025): Discovers quality features from data. Complements CPGA: AutoQual discovers what to check; FORGE determines how to enforce it at the cheapest layer.

9. Implementation Advances

This section documents architectural advances developed during production implementation that extend the core primitives validated in §6.

9.1 Section-Wise Rule Extraction

The classification protocol (§3.3) assumes a set of atomic context items as input. In practice, these items must be extracted from natural-language source documents. Sending an entire source document (~40K characters) to a single LLM extraction call suffers from the same context degradation (F1, F2) that CPGA addresses in generation. Rules described in the middle of the document are systematically under-extracted.

Solution: Split each source document by heading into sections. Extract context items per section in separate LLM calls (~2–5K characters each). The section heading provides a natural scope signal — items extracted from a section titled "Security Policy" are tagged with scope security without requiring the LLM to infer scope from the full document. Cross-section deduplication removes items that appear in multiple sections (e.g., a preamble mentions a rule that a later section details). The more specific version is retained.

This is CPGA applied to its own setup phase: partition the extraction context to avoid the attention decay that would cause rule loss.

9.2 Evidence-Weighted CADG

The CADG mechanism (§4.3) permutes constraint order uniformly at random. This de-correlates position-bias failures but treats all constraints equally. In practice, some constraints fail far more often than others. Placing a rarely-violated constraint in the primacy slot wastes the attention benefit.

Solution: Weight the permutation by historical failure rate. Candidate 0 (the baseline) sorts constraints by failure rate descending — the hardest constraints occupy the primacy slot. Candidates 1+ use weighted random sampling where high-failure constraints have proportionally higher probability of early placement, with sufficient randomness to maintain de-correlation across candidates.

The failure weights are loaded from a global history file that accumulates pass/fail data across all production runs. The weights are computed once per plan build (not per candidate) and cached. When no history exists (cold start), the system falls back to uniform permutation.

Cost: Zero — same number of LLM calls, same token count. Only the ordering changes. The improvement is concentrated where it matters most.

9.3 Sentinel Feedback Loop to FORGE

FORGE classifies rules once at setup time (§3.3). But sentinel pass/fail data collected during production reveals information that FORGE's initial classification cannot: which Layer 3 rules the model follows without prompting.

When a rule is classified as Layer 3 (residual prompt) but the sentinel for that rule consistently passes even in generations where the rule was not in the prompt (because it was outside the scope partition for that section), this is strong evidence that the rule can be promoted to Layer 1. The model follows it naturally, or the compiler/IR schema enforces it implicitly.

Mechanism: After each sentinel swarm run, compare the set of sentinel verdicts against the set of Layer 3 rule IDs that were actually in the prompt. Rules that passed but were absent from the prompt get an unprompted_pass counter incremented. When a rule reaches 95%+ unprompted pass rate over 10+ observations, it is surfaced as a promotion candidate.

This closes the feedback loop between SENTINEL (§5) and FORGE (§3.3), enabling data-driven layer reclassification without re-running the probe protocol.

9.4 Progressive Promotion Protocol

The progressive promotion mechanism (§3.4) is validated in production through the sentinel feedback loop (§9.3). See §3.4 for the full protocol; production validation confirms that the prompt shrinks measurably with each batch as rules accumulate sufficient unprompted-pass evidence for Layer 3→1 promotion.

9.5 IR-Level Mutation (Structural Regression Prevention)

When sentinel verdicts indicate failures requiring content-level changes, the mutation agent operates on the Intermediate Representation (typed JSON), not the final output. The deterministic compiler re-renders the corrected IR to final format. This architectural choice prevents a common failure mode in LLM-based repair: the repair model "fixes" a content issue but introduces structural malformation (broken tags, missing required elements, format violations).

Because the compiler enforces structural invariants regardless of IR content, structural regressions from content repairs are architecturally impossible. The mutation plan provides each failing IR chunk alongside the corresponding sentinel verdicts, so the repair agent receives full context of what failed and why — but can only modify the content fields of the typed schema. The compiler guarantees that the re-rendered output satisfies all structural constraints, regardless of what the repair agent changes in the IR.

This decoupling — content mutations in IR-space, structural guarantees in compiler-space — means the repair loop can be aggressive (large content changes, full section rewrites) without risking the structural compliance that Tier 2/3 sentinels enforce. In practice, this eliminates the "fix-break" cycle where repairing one constraint violation introduces another.

Limitations

Several limitations scope the claims of this work. First, model dependence: all experiments use frontier commercial models (Claude Opus 4.6, GPT-5.4) accessed via hosted APIs; results may differ on smaller or open-weight models where attention behavior and instruction-following capabilities vary. Second, Layer 0 availability: token-mask enforcement requires logit-level or constrained-decoding API access, which is not universally available across all providers; hosted-API deployments without such access must reclassify Layer 0 items to Layer 1. Third, Tier 4 cost at scale: semantic sentinels (micro-LLM calls) add per-rule latency and cost proportional to the number of Layer 4 rules; for very large rule sets (1,000+), the cost of parallel Tier 4 calls becomes non-trivial. Fourth, English-only evaluation: all benchmarks and production evaluation are English-language; cross-lingual generalization is untested. Fifth, custom benchmark caveat: SysPrompt is designed for this work and will be released with the code; results on this benchmark should be interpreted with appropriate caution pending independent replication. Finally, production evaluation is observational: the production deployment (§6.10) reports before/after metrics without a controlled A/B test or blind evaluation; the +14.3pp improvement reflects the full system redesign (FORGE + IR compiler + SENTINEL), not an isolated ablation of any single component.

Ethical Considerations

This work proposes an architectural framework for improving LLM instruction-following compliance. The architecture is domain-agnostic and does not introduce new capabilities for generating harmful content. The enforcement mechanisms (token masks, compiled checkers, external sentinels) constrain LLM outputs to comply with specified rules, which in practice increases safety by enabling more reliable adherence to content policies and behavioral guidelines. All experiments use publicly available benchmark data or custom benchmarks that will be released with the code. No human subjects were involved. The production deployment described in §6.10 is used for structured content generation in a commercial setting; no personally identifiable information is processed. API costs for the full experimental suite were approximately $15 for controlled experiments and $200 for benchmark evaluation, using standard commercial pricing.

10. Conclusion

We have presented CPGA+SENTINEL, an architecture that addresses LLM context degradation through a single organizing principle: classify every context item by whether it genuinely requires the LLM's attention, and externalize everything that does not. The FORGE classification protocol provides a systematic, automated mechanism for this classification — replacing the ad hoc, manual decisions that characterize current production systems. CADG exploits the known position-bias failure mode as a diversity mechanism, yielding consistent gains at zero additional cost within the multi-candidate budget. The SENTINEL swarm externalizes enforcement into independent per-rule agents that serve a dual role as both production enforcement and evaluation infrastructure.

Three findings are particularly significant. First, component composition is benchmark-dependent: the optimal combination varies with constraint taxonomy characteristics (flat lists favor CADG+SENTINEL; hierarchical rule sets with many code-checkable rules favor the full stack including FORGE). This argues against monolithic architectures and for principled composition guided by constraint profiling. Second, the retry-with-feedback baseline is surprisingly weak — achieving only +0.5pp on MOSAIC despite having access to the same deterministic checkers and explicit failure feedback. Architectural partitioning consistently outperforms compute-scaling approaches across all four benchmarks. Third, the gap between controlled experiments and production deployment is informative: FORGE's modest +0.9pp on MOSAIC's flat constraint lists versus +14.3pp contribution to the production system (577 hierarchical rules) demonstrates that the architecture's value scales with the complexity of the rule taxonomy it manages.

The SENTINEL dual-role architecture — where the same swarm infrastructure serves as both enforcement mechanism and evaluation suite — has a practical consequence: deploying CPGA+SENTINEL for production automatically creates a regression testing framework that grows with the enforcement suite. This closes a gap in current LLM deployment practice, where enforcement and evaluation are typically separate systems maintained by different teams.
